## 一、系统总体设计

### 1.1 需求分析

系统需要解决的核心问题：**帮助视障人士通过语音指令，引导手部准确移动到目标物体位置并抓取**。

关键约束条件：

| 约束 | 要求 |
|------|------|
| **实时性** | 语音指令延迟需控制在可接受范围内（< 500ms 感知延迟） |
| **准确性** | 物体检测和手部追踪精度足以生成有效指令 |
| **全方向导航** | 需覆盖三维六方向（上/下/左/右/前/后），不仅是二维平面 |
| **可用性** | 语音输出清晰、不轰炸用户、有防抖机制 |
| **硬件限制** | 仅使用手机摄像头 (DroidCam) + 普通电脑 CPU，无专用硬件 |

### 1.2 架构设计思路

系统采用 **前后端分离 + WebSocket 实时通信** 架构：

```
┌─────────────────────────────────────────────────────┐
│                 后端 Python (FastAPI)                 │
│                                                     │
│  DroidCam ──→ FrameGrabber ──→ 推理管线 ──→ JSON   │
│  (MJPEG)     (后台线程)      (ThreadPool)           │
│                                                     │
│  推理管线内部:                                       │
│  旋转 → 缩放 → YOLO → MediaPipe → MiDaS → 指令     │
│       → 防抖 → 编码 → WebSocket 推送                │
└──────────────────┬──────────────────────────────────┘
                   │ WebSocket (JSON)
┌──────────────────▼──────────────────────────────────┐
│                 前端 React (Vite)                     │
│                                                     │
│  WebSocket ──→ 画面渲染 + 指令显示 + TTS 播报        │
│  前端 ────→ 目标选择 / 旋转控制 (WS 消息上行)        │
└─────────────────────────────────────────────────────┘
```

**为什么选择 WebSocket 而非 HTTP 轮询？**

- 视频流 + 指令需要持续推送，HTTP 轮询的请求/响应开销过大
- WebSocket 全双工连接可同时传输画面和接收用户指令
- 单连接服务二进制帧 + JSON 结构化数据

**为什么后端拉取 DroidCam 而非前端直接播放？**

- 后端需要对每帧执行 YOLO / MediaPipe / MiDaS 推理
- 如果前端采集后上传，会增加一次传输延迟
- 后端直接从 DroidCam HTTP 拉取 MJPEG 流是最短路径

### 1.3 模块划分

从最初的单文件 `main.py`（~821 行）重构为 6 个职责明确的模块：

| 模块 | 行数 | 职责 | 依赖 |
|------|------|------|------|
| `config.py` | ~76 | 所有全局常量 + 可变运行时状态 | 无 |
| `models.py` | ~70 | 模型加载（YOLO / MediaPipe / MiDaS） | config |
| `vision.py` | ~420 | 视觉处理（帧操作、检测、绘制、深度） | config, models |
| `guidance.py` | ~190 | 指令生成 + 防抖冷却 | config, vision |
| `routes.py` | ~280 | HTTP + WebSocket 路由 | config, models, vision, guidance |
| `main.py` | ~45 | FastAPI 入口 | models, routes |

依赖关系（单向，无循环）：

```
config ← models ← vision ← guidance ← routes ← main
```

### 2.3 vision.py — 视觉处理层

该模块包含约 420 行代码，涵盖帧采集、编解码、绘制、检测、深度估计全部视觉功能。

#### 2.3.1 FrameGrabber 类

**解决的问题**：OpenCV `VideoCapture.read()` 从 DroidCam MJPEG 流读取时，会积压 2-3 秒的缓冲帧。调用 `cap.read()` 读到的不是当前帧，而是队列中最老的帧，导致严重延迟。

**解决方案**：独立后台线程持续 `read()` 丢弃旧帧，只保留最新一帧。

```python
class FrameGrabber:
    def __init__(self, url: str):
        self._cap = cv2.VideoCapture(url)
        self._cap.set(cv2.CAP_PROP_BUFFERSIZE, 1)  # 最小化 OpenCV 内部缓冲
        self._lock = threading.Lock()
        self._frame = None                           # 只存最新一帧
        self._running = False

    def _grab_loop(self):
        """后台线程：持续读取，每次覆盖上一帧"""
        while self._running:
            ret, frame = self._cap.read()
            if ret and frame is not None:
                with self._lock:
                    self._frame = frame  # 新帧直接覆盖旧帧

    def read(self):
        """主线程调用：取走最新帧，标记已消费"""
        with self._lock:
            frame = self._frame
            self._frame = None  # 避免重复处理同一帧
            return frame
```

**设计要点**：

- `threading.Lock` 确保 `_frame` 的读写线程安全
- `read()` 返回后将 `_frame` 置 `None`，如果处理速度快于采集，主循环会短暂等待而非重复处理同一帧
- `CAP_PROP_BUFFERSIZE = 1` 进一步减少 OpenCV 底层缓冲
- `daemon=True` 线程随主进程退出自动销毁

#### 2.3.2 帧编解码与旋转

```python
def resize_frame(frame, target_width=640):
    """保持宽高比缩放到指定宽度"""
    h, w = frame.shape[:2]
    if w <= target_width:
        return frame
    scale = target_width / w
    new_h = int(h * scale)
    return cv2.resize(frame, (target_width, new_h), interpolation=cv2.INTER_LINEAR)
```

- 使用 `INTER_LINEAR` 双线性插值，在速度和质量之间取平衡
- 只有当原始宽度大于目标值时才缩放，避免放大模糊

```python
def encode_frame(frame):
    """以配置的 JPEG 质量压缩并 Base64 编码"""
    _, buffer = cv2.imencode('.jpg', frame,
                              [cv2.IMWRITE_JPEG_QUALITY, config.JPEG_QUALITY])
    return f"data:image/jpeg;base64,{base64.b64encode(buffer).decode('utf-8')}"
```

- `config.JPEG_QUALITY = 40`：经测试，40 的 JPEG 压缩在视频流中几乎无可见质量损失，但文件大小仅为 quality=90 时的 ~30%
- Base64 编码后直接嵌入 `data:image/jpeg;base64,` 前缀，前端 `<img src>` 可以直接使用

#### 2.3.4 深度估计

```python
def estimate_depth(frame):
    img_rgb = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
    input_batch = depth_transform(img_rgb).to('cpu')

    with torch.no_grad():
        prediction = depth_model(input_batch)

    # 插值回原图尺寸
    prediction = torch.nn.functional.interpolate(
        prediction.unsqueeze(1),
        size=frame.shape[:2],
        mode="bicubic",
        align_corners=False,
    ).squeeze()

    depth_map = prediction.cpu().numpy()

    # 归一化到 [0, 1]
    d_min, d_max = depth_map.min(), depth_map.max()
    depth_map = (depth_map - d_min) / (d_max - d_min)
    return depth_map
```

**关键细节**：

1. **`torch.no_grad()`**：禁用梯度计算，推理时节省 ~30% 内存和计算
2. **`interpolate` 回原图尺寸**：MiDaS 内部将图像缩放到固定尺寸推理，输出需要插值回原始分辨率，才能用像素坐标查询深度
3. **`bicubic` 插值**：深度图是连续值，双三次插值比最近邻更平滑
4. **归一化到 [0, 1]**：MiDaS 输出的是相对深度值（无单位），归一化后 0=最近，1=最远
5. **`get_depth_at()` 使用 10×10 patch 平均**：单像素深度值噪声大，取邻域均值更稳定

### 2.4 guidance.py — 指令生成层

#### 2.4.1 generate_instruction — 六方向导航

这是系统核心算法，将目标物体和手部的三维空间关系映射为六方向语音指令。

```python
def generate_instruction(target, hand, target_name, threshold,
                         depth_map=None, depth_threshold=0.03):
```

**算法步骤**：

```
第一层判断：Z轴（前后方向），使用深度图
    ├── depth_hand - depth_obj > 0.03  →  "Move Forward"  (手离相机远，物体离相机近)
    ├── depth_hand - depth_obj < -0.03 →  "Move Backward" (手离相机近，物体离相机远)
    └── |差值| ≤ 0.03                  →  进入第二层判断

第二层判断：X/Y轴（平面方向），使用像素坐标
    ├── |dx| ≤ threshold 且 |dy| ≤ threshold  →  "Grasp"
    ├── |dx| ≥ |dy|（水平偏移更大）
    │   ├── dx > threshold   →  "Move Right"   (目标在手的右边)
    │   └── dx < -threshold  →  "Move Left"    (目标在手的左边)
    └── |dy| > |dx|（垂直偏移更大）
        ├── dy > threshold   →  "Move Down"    (目标在手的下方)
        └── dy < -threshold  →  "Move Up"      (目标在手的上方)
```

**设计决策 — Z轴优先于X/Y轴**：

- 在实际使用中，如果手和物体不在同一深度平面，单纯的左右上下指令会误导用户
- 例如：手在物体正上方但远离相机 30cm，此时应该先说 "前进"，而非 "向下"
- 深度对齐后再进行平面导航，最终收敛到 "Grasp"

**`dx` 和 `dy` 的计算方向**：

```python
dx = target["cx"] - hand["cx"]   # 正值 = 目标在手的右边
dy = target["cy"] - hand["cy"]   # 正值 = 目标在手的下方（图像坐标系 y 轴向下）
```

#### 2.4.2 stabilize_instruction — 防抖 + 冷却

**解决的问题**：

1. **帧间抖动**：由于检测精度限制，连续帧可能在 "Move Left" 和 "Move Right" 之间频繁切换
2. **语音轰炸**：每帧都播报指令会导致 TTS 堆叠，用户听不清

**两层过滤机制**：

```
原始指令 (每帧一个)
    │
    ├── 防抖过滤：连续 10 帧 (STABLE_THRESHOLD) 相同指令才视为"稳定"
    │   不满足 → 不发送（返回空字符串）
    │   满足 ↓
    │
    ├── 冷却过滤：与上一次发送的指令对比
    │   ├── 指令内容变了 → 立即发送
    │   └── 指令内容相同 → 距上次 > 3 秒 (COOLDOWN_SECONDS) 才重发
    │
    └── 输出：需要语音播报的指令（空字符串 = 不播报）
```

```python
def stabilize_instruction(raw_instruction):
    # 防抖计数
    if raw_instruction == config.last_raw_instruction:
        config.stable_instruction_count += 1
    else:
        config.stable_instruction_count = 1
        config.last_raw_instruction = raw_instruction

    instruction_to_send = ""

    # 达到稳定阈值
    if config.stable_instruction_count >= config.STABLE_THRESHOLD:
        if raw_instruction != config.last_sent_instruction:
            # 指令变了，立即发送
            instruction_to_send = raw_instruction
            config.last_sent_instruction = raw_instruction
            config.last_sent_time = current_time
        else:
            # 指令未变，检查冷却
            if current_time - config.last_sent_time > config.COOLDOWN_SECONDS:
                instruction_to_send = raw_instruction
                config.last_sent_time = current_time

    return instruction_to_send
```

#### 2.5.2 `_process_frame_sync()` — 核心推理管线

这是单帧处理的同步函数，所有 CPU 密集型计算封装在此，通过 `ThreadPoolExecutor` 在工作线程中执行。

```python
def _process_frame_sync(frame):
    # 1. 旋转画面
    if config.ROTATION != 0:
        frame = rotate_frame(frame, config.ROTATION)

    # 2. 缩放到处理宽度
    frame = resize_frame(frame, target_width=config.PROCESS_WIDTH)

    # 3. YOLO 物体检测（按间隔执行 + 缓存）
    config.frame_count += 1
    if config.frame_count % config.YOLO_INTERVAL == 1 or config.yolo_cache is None:
        config.yolo_cache = yolo_model(frame, verbose=False, device='cpu')
    draw_yolo_detections(frame, config.yolo_cache)

    # 4. 目标锁定
    target_info = find_best_target(config.yolo_cache, config.TARGET_OBJECT)
    if target_info:
        draw_target_highlight(frame, target_info)

    # 5. MediaPipe 手部检测（每帧执行）
    rgb_frame = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
    rgb_frame.flags.writeable = False    # MediaPipe 性能优化
    hand_results = hands.process(rgb_frame)
    draw_hand_landmarks(frame, hand_results)
    hand_center = compute_hand_center(hand_results, w, h)

    # 6. MiDaS 深度估计（按间隔执行 + 缓存）
    if (config.DEPTH_INTERVAL > 0
            and config.frame_count % config.DEPTH_INTERVAL == 1):
        config.depth_cache = estimate_depth(frame)

    # 7. 指令生成
    raw_instruction = generate_instruction(target_info, hand_center, ...)

    # 8. 防抖 + 冷却
    instruction_to_send = stabilize_instruction(raw_instruction)

    # 9. 检测物体列表（去重）
    all_objects = get_all_detections(config.yolo_cache)

    # 10. JPEG 编码
    encoded = encode_frame(frame)

    return {
        "image": encoded,
        "instruction": raw_instruction,
        "display_instruction": raw_instruction,
        "speech_instruction": instruction_to_send,
        "detected_objects": [obj["name"] for obj in all_objects],
        "target": config.TARGET_OBJECT,
    }
```

**`rgb_frame.flags.writeable = False` 的作用**：
MediaPipe 文档指出，将输入标记为不可写可以避免内部拷贝，提升 ~10% 性能。

#### 2.5.3 WebSocket `/ws/video` — 主端点

```python
@router.websocket("/ws/video")
async def websocket_video(websocket: WebSocket):
    await websocket.accept()

    # 1. 启动 FrameGrabber
    grabber = FrameGrabber(config.DROIDCAM_URL)
    grabber.start()

    # 2. 启动后台消息接收协程
    recv_task = asyncio.create_task(_ws_receive_loop(websocket))

    # 3. 主循环：取帧 → 推理 → 推送
    try:
        while True:
            frame = grabber.read()
            if frame is None:
                await asyncio.sleep(0.01)
                continue

            # 在线程池中执行同步推理（不阻塞事件循环）
            result = await loop.run_in_executor(
                _inference_executor, _process_frame_sync, frame
            )

            await websocket.send_json(result)
            await asyncio.sleep(1.0 / config.TARGET_FPS)
    finally:
        recv_task.cancel()
        grabber.stop()
```

**三大并发机制**：

```
┌─ 后台线程: FrameGrabber._grab_loop() [持续读取 DroidCam]
│
├─ 工作线程: _inference_executor [执行 _process_frame_sync]
│
├─ 协程 1: websocket_video() 主循环 [取帧 → 调度推理 → 推送]
│
└─ 协程 2: _ws_receive_loop() [接收前端消息：目标/旋转切换]
```

**`_ws_receive_loop` 为什么需要单独协程？**

- WebSocket 是全双工的，接收和发送互不阻塞
- 如果在主循环中穿插 `receive()`，必须处理无消息时的超时逻辑，代码复杂
- 单独协程 + `asyncio.create_task()` 更清晰，`_ws_receive_loop` 内部 `await websocket.receive_json()` 是零开销挂起

#### 2.5.4 `ThreadPoolExecutor(max_workers=1)` 的设计

```python
_inference_executor = ThreadPoolExecutor(max_workers=1)
```

为什么是单线程？

- YOLO、MediaPipe、MiDaS 都不是线程安全的（共享 GPU 缓冲区、内部状态）
- 单线程确保同一时刻只有一个帧在推理，避免竞态条件
- 它的作用是**不阻塞 asyncio 事件循环**，而非并行推理

### 3.3 TTS 语音播报

```jsx
const speak = useCallback((text) => {
    if (!text || text === lastCommandRef.current) return;  // 防重复
    lastCommandRef.current = text;
    window.speechSynthesis.cancel();  // 取消正在播放的语音

    // 创建两个 utterance，每条指令读两遍
    const utt1 = new SpeechSynthesisUtterance(text);
    utt1.lang = 'en-US';
    utt1.rate = 1.1;
    const utt2 = new SpeechSynthesisUtterance(text);
    utt2.lang = 'en-US';
    utt2.rate = 1.1;

    window.speechSynthesis.speak(utt1);
    window.speechSynthesis.speak(utt2);
}, []);
```

**为什么每条指令读两遍？**

- 视障用户在对环境注意力分散的情况下，第一遍可能没听清
- 读两遍的总时长约 1.5 秒，在 3 秒冷却周期内刚好合适

**为什么用 `window.speechSynthesis.cancel()` 先清空队列？**

- 如果新指令到来时旧语音未播完，直接追加会造成播放积压
- `cancel()` 立即停止当前语音并清空队列

**`rate = 1.1`**：略快于正常语速，在给用户充足信息的同时减少等待感。

## 四、核心算法详解

### 4.1 物体检测 — YOLOv8

YOLOv8 (You Only Look Once v8) 是 Ultralytics 2023 年发布的实时物体检测模型。

**调用方式**：

```python
results = yolo_model(frame, verbose=False, device='cpu')
```

- `verbose=False` 关闭逐帧日志（否则每帧打印大量信息）
- `device='cpu'` 强制 CPU 推理

**输出结构**：

```python
results[0].boxes          # 所有检测到的边界框
results[0].boxes.xyxy     # 坐标 [x1, y1, x2, y2]
results[0].boxes.conf     # 置信度 [0, 1]
results[0].boxes.cls      # 类别 ID
results[0].names          # 类别 ID → 名称映射 {0: 'person', 1: 'bicycle', ...}
```

**预训练类别**：COCO 数据集 80 类，包括 person, bicycle, car, motorcycle, airplane, bus, train, truck, boat, traffic light, fire hydrant, stop sign, parking meter, bench, bird, cat, dog, horse, sheep, cow, elephant, bear, zebra, giraffe, backpack, umbrella, handbag, tie, suitcase, frisbee, skis, snowboard, sports ball, kite, baseball bat, baseball glove, skateboard, surfboard, tennis racket, bottle, wine glass, cup, fork, knife, spoon, bowl, banana, apple, sandwich, orange, broccoli, carrot, hot dog, pizza, donut, cake, chair, couch, potted plant, bed, dining table, toilet, tv, laptop, mouse, remote, keyboard, cell phone, microwave, oven, toaster, sink, refrigerator, book, clock, vase, scissors, teddy bear, hair drier, toothbrush.

### 4.2 手部追踪 — MediaPipe Hands

MediaPipe Hands 是 Google 开发的轻量级手部关键点检测模型，输出 21 个 3D 关键点。

**21 个关键点布局**：

```
              8   12  16  20
              |   |   |   |
          7   11  15  19
          |   |   |   |
      6   10  14  18
      |   |   |   |
  5   9   13  17
  |   |   |   |
  4   3   2   1
   \  |  /   /
    \ | /   /
     \|/   /
      0 (WRIST)
```

关键点：0=手腕, 4=拇指尖, 8=食指尖, 12=中指尖, 16=无名指尖, 20=小指尖

**手部中心计算**：

```python
cx = sum(p.x for p in landmarks) / 21 * frame_w
cy = sum(p.y for p in landmarks) / 21 * frame_h
```

取全部 21 个点的质心，比取单个关键点（如手腕或中指根部）更能代表"手掌区域"的位置。

### 4.3 深度估计 — MiDaS

MiDaS (Monocular Depth estimation in the wild) 是 Intel ISL 的单目深度估计模型。从单张 RGB 图像中推断每个像素距离相机的相对远近。

**MiDaS 输出特性**：

- 输出是**相对深度**（不是绝对距离），值越大表示离相机越远
- 归一化后范围 [0, 1]，0=最近，1=最远
- **同一帧内**不同像素的深度值可以比较，但不同帧之间的绝对值不可比

**深度查询**：

```python
def get_depth_at(depth_map, cx, cy, patch_size=10):
    """取 10×10 邻域平均深度"""
```

使用 10×10 patch 均值而非单像素：
- 深度图有噪声，边缘区域尤为明显
- 10×10 约覆盖 1cm² 真实面积（取决于距离），作为"点"已足够精确

### 4.4 指令生成 — 六方向空间导航

完整的指令状态机：


## 五、性能优化策略

原始版本存在 2-3 秒的可感知延迟，经过以下五层优化后降至可接受范围。

### 5.1 FrameGrabber — 零延迟帧获取

**问题**：OpenCV `VideoCapture` 内部维护了一个多帧缓冲队列。当推理速度（~150ms/帧）慢于 DroidCam 推送速率（30fps = 33ms/帧）时，缓冲区持续增长，主循环 `cap.read()` 取到的帧总是数秒前的旧帧。

**方案**：后台守护线程持续 `cap.read()` 消耗缓冲区，只保留最新一帧。

**效果**：从直接 `cap.read()` 的 2-3 秒延迟降至接近 0 延迟（帧年龄 < 推理耗时）。

### 5.2 ThreadPoolExecutor — 非阻塞推理

**问题**：YOLO + MediaPipe + MiDaS 推理全部在 asyncio 事件循环线程中执行（`_process_frame_sync` 是同步函数），阻塞了 WebSocket 的消息收发、心跳等异步操作。

**方案**：

```python
_inference_executor = ThreadPoolExecutor(max_workers=1)

result = await loop.run_in_executor(_inference_executor, _process_frame_sync, frame)
```

**效果**：推理在工作线程中执行，事件循环线程保持响应。WebSocket 消息收发不再被推理阻塞。

### 5.3 推理间隔与缓存

| 模型 | 帧间隔 | CPU 耗时/次 | 说明 |
|------|--------|------------|------|
| YOLO | 每 3 帧 | ~130ms | 物体位置变化慢，3 帧内几乎不变 |
| MediaPipe | 每帧 | ~15ms | 手部移动快，需实时追踪 |
| MiDaS | 每 15 帧 | ~200ms | 深度变化最慢，高开销 |

**缓存策略**：

```python
# YOLO 缓存
if config.frame_count % config.YOLO_INTERVAL == 1 or config.yolo_cache is None:
    config.yolo_cache = yolo_model(frame, ...)
# 其余帧复用 yolo_cache

# MiDaS 缓存
if config.frame_count % config.DEPTH_INTERVAL == 1:
    config.depth_cache = estimate_depth(frame)
# 其余帧复用 depth_cache
```

**效果**：平均每帧推理时间从 ~345ms 降至 ~85ms（YOLO 130/3 + MediaPipe 15 + MiDaS 200/15 ≈ 72ms，加开销 ≈ 85ms）。

### 5.4 分辨率与编码优化

| 参数 | 优化前 | 优化后 | 影响 |
|------|--------|--------|------|
| 处理分辨率 | 640px | 480px | 推理速度提升 ~44%（面积减少 44%） |
| JPEG Quality | 默认 95 | 40 | 传输量减少 ~70%，肉眼几乎无差别 |

**为什么 480px 而不是更低？**

- 320px 下 YOLO 小物体检测率显著下降（小物体完全消失）
- 480px 是检测精度几乎不受影响的最低阈值

### 5.5 帧率限制

```python
await asyncio.sleep(1.0 / config.TARGET_FPS)  # 1/15 ≈ 66ms
```

- 限制 WebSocket 推送频率为 15fps
- 人眼对视频流感知的流畅阈值约 12-15fps
- 避免推理比显示快时前端消息堆积

## 六、数据流与时序分析

完整的单帧处理时序（优化后）：

```
时间轴 ────────────────────────────────────────────────────→

后台线程 (FrameGrabber):
  cap.read() → cap.read() → cap.read() → cap.read() → ...
  [持续以 30fps 读取并丢弃旧帧，保留最新帧]

主协程 (websocket_video):
  T₀: frame = grabber.read()
  T₁: result = await run_in_executor(process_frame)  ←─ 非阻塞等待
  T₁': [事件循环可处理其他协程：_ws_receive_loop 等]
  T₂: await websocket.send_json(result)
  T₃: await asyncio.sleep(66ms)                      ←─ 帧率限制
  T₄: 回到 T₀

工作线程 (_process_frame_sync):
  T₁ → T₂ (约 85ms):
    旋转 (1ms)
    → 缩放 (2ms)
    → YOLO (仅 1/3 帧执行: 130ms, 否则 0ms)
    → 绘制 YOLO (3ms)
    → 目标锁定 (1ms)
    → MediaPipe (15ms)
    → 绘制手部 (2ms)
    → 计算手心 (0.1ms)
    → MiDaS (仅 1/15 帧执行: 200ms, 否则 0ms)
    → 指令生成 (0.1ms)
    → 防抖过滤 (0.1ms)
    → 物体列表 (0.5ms)
    → JPEG 编码 (5ms)

前端 (App.jsx):
  ws.onmessage:
    JSON.parse (1ms)
    → setDisplayImage (React render ~5ms)
    → speak() if speech_instruction (TTS 异步队列)

后台协程 (_ws_receive_loop):
  持续等待 websocket.receive_json()
  用户点击目标 → {"target": "bottle"} → config.TARGET_OBJECT = "bottle"
  用户点击旋转 → {"rotation": 180} → config.ROTATION = 180
```

**端到端延迟分析**：

| 阶段 | 延迟 |
|------|------|
| DroidCam → FrameGrabber | ~0ms（后台线程连续读取） |
| FrameGrabber → 工作线程 | ~0ms（取最新帧） |
| 推理管线 | ~85ms（平均） |
| WebSocket 传输 | ~5ms（本地） |
| 前端渲染 | ~5ms |
| 帧率等待 | ~66ms |
| **总计** | **~160ms** |

## 七、关键设计决策与理由

### 为什么选择 FastAPI + uvicorn 而非 Flask？

- FastAPI 原生支持 WebSocket（`@router.websocket`），Flask 需要 `flask-socketio` 等额外依赖
- `async/await` 协程模型适合 I/O 密集型场景（WebSocket 持续推送）
- uvicorn 的 `websockets` 后端性能优于 Socket.IO 的轮询 fallback

### 为什么使用 DroidCam 而非 USB 摄像头？

- 模拟视障场景中更可能使用的移动端设备
- DroidCam 提供标准 MJPEG HTTP 流，OpenCV 可直接读取
- 手机摄像头质量通常优于笔记本内置摄像头

### 为什么选择 MiDaS Small 而非其他深度模型？

| 方案 | CPU 推理耗时 | 说明 |
|------|-------------|------|
| MiDaS DPT-Large | >1000ms | 过慢，不适合实时 |
| MiDaS DPT-Hybrid | ~600ms | 仍然太慢 |
| **MiDaS Small** | **~200ms** | **最快的 MiDaS 变体** |
| 立体视觉 | N/A | 需要双目相机硬件 |
| ToF 传感器 | N/A | 需要专用硬件 |

MiDaS Small 是在仅有单目 RGB 输入 + CPU 约束下的最佳选择。

### 为什么指令采用英文而非中文？

- Web Speech API 的英文 TTS 质量和流畅度显著优于中文
- 英文指令更短（"Move Left" 2个词 vs "请向左移动" 4个字），播报更快
- 便于跨语言用户使用

### 为什么用 Base64 传输图片而非 Binary Frame？

- JSON 消息中同时包含 image + instruction + detected_objects 等结构化字段
- 如果用 Binary Frame，需要自定义帧协议分离二进制图片和元数据
- Base64 编码增加 ~33% 体积，但在本地网络中对延迟的影响可忽略
- 浏览器 `<img src="data:image/jpeg;base64,...">` 直接渲染，无需 Blob URL 管理

### 为什么防抖阈值是 10 帧？

- 在 15fps 下，10 帧 = 0.67 秒
- 小于 0.5 秒的抖动（如检测框闪烁）被过滤
- 用户真实的手部移动持续时间远超 0.67 秒
- 该值在实测中在"响应速度"和"抖动过滤"之间取得最佳平衡

### 为什么冷却时间是 3 秒？

- TTS 读两遍一条指令约需 1.5 秒
- 3 秒冷却 = 读完上一条后有 1.5 秒静默，供用户消化信息
- 过短会造成语音重叠，过长会让用户以为系统无响应
